## 19:33 - Wingdk 视频定格Bug根因分析

### 问题现象
Wingdk 成品视频（326.8MB）画面在 **7分55秒** 定格，但音频继续播了将近4分钟（视频流474.9s vs 音频流706.1s）。

### 根因链（Bug Chain）

**Bug 1 - seg_02 moov atom 损坏：**
- 截取 seg_02 后文件还在写入中就被 normalization loop 开始读取
- ffmpeg 报告 "moov atom not found"，文件损坏（0KB 或不完整）
- 音频重编码（step 2.3）静默跳过该文件（不重编码，保留原文件）
- 但损坏的视频轨道（视频流长度不一致）被直接传下去

**Bug 2 - xfade 使用损坏文件失败，回退 concat：**
- xfade filter_complex 执行时，seg_02.mp4 被检测为损坏 → xfade 失败
- 回退到简单 concat，但 concat list 包含 seg_02（损坏文件）
- ffmpeg concat `-c copy` 静默跳过损坏的 video stream，只拼接了 seg_00 + seg_01
- 输出 concat_xfade.mp4 **视频流只有 seg_00+01 的长度**（约475秒）

**Bug 3 - Step 6/7 混音用 `-c copy` 保持破损视频：**
- Step 6: `ffmpeg -i concat_xfade.mp4 -i merged_commentary.mp3 -c copy` 
- concat -c copy 把截断的视频流直接复制 → 最终成品 video track = 474.9秒
- 音频是完整的 TTS 混音 = 706.1秒
- 结果：画面在 7:55 定格，音频继续播放

### 关键日志节点
| 时间 | 事件 |
|------|------|
| 17:59:18 | seg_02 音频重编码失败（moov atom not found）|
| 18:07:25 | xfade 失败（seg_02.mp4 损坏）|
| 18:07:26 | 回退 concat -c copy（静默丢弃损坏片段）|
| 18:11:27 | Step 7 混音保持截断视频流 |
| 成品 | 视频474.9s / 音频706.1s = 画面定格 |

### 三个具体Bug
1. **文件句柄竞争**（根因）：截取文件写完前被 normalization 开始读 → moov atom 不完整
2. **concat -c copy 静默失败**：损坏文件视频流被跳过，不报错 → 截断视频流
3. **Step 6 concat -c copy 不校验时长**：用 stream copy 不重新编码，无法发现视频被截断

### 修复方案（已在进行）
- Step 6 concat 后加**时长校验**：比较视频流实际时长 vs 预期时长，不吻合则重新编码视频（不用 -c copy）
- 截取和标准化之间加**文件完整性等待**：所有文件写完/句柄释放后再进入标准化
